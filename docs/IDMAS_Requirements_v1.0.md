## 二、背景与目标 【必须】

### 2.1 项目背景

2026年被业界定义为企业级多智能体规模化"上岗元年"。工业制造领域正从"流程驱动"向"智能驱动"转型，多Agent系统通过专业化分工实现跨部门协作，覆盖设计、质检、运维、工艺规划等核心环节。三菱电机等行业龙头已部署基于对抗辩论的多Agent AI，用于生产规划、安全分析与风险评估。

**当前核心痛点：**

| # | 痛点 | 影响 |
|---|------|------|
| P1 | 识图依赖专家经验 | 专利图纸、机械结构图的标号识别高度依赖资深工程师，知识传承困难 |
| P2 | 文本与图纸割裂 | PLM/MES系统中图纸与说明书分离，工程师需多系统切换核对 |
| P3 | 跨部门协同低效 | 设计/工艺/质检/运维对同一张图纸的理解维度不同，缺乏统一智能解析中枢 |
| P4 | 知识沉淀困难 | 图纸解析结论分散在个人电脑或邮件中，无法形成可检索的企业知识资产 |

### 2.2 立项原因

- 企业级多智能体技术成熟度已达到工业落地水平
- 开源 VLM（Qwen2.5-VL / InternVL2.5）视觉理解能力已达工业图纸识别可用水平，大模型蒸馏+全量微调路线可行
- 行业对图纸智能解析的需求紧迫，竞品覆盖有限，存在差异化窗口期

### 2.3 业务目标 OKR

**Objective：构建以工业图纸视觉识别Agent为核心的多Agent协同系统，实现人机协同识图**

| Key Result | 目标值 | 衡量方式 |
|------------|--------|---------|
| KR1：Vision Agent 标号识别准确率（ft_strict） | > 0.55（及格）/ > 0.70（优秀） | eval_dual_track.py 每轮训练后评估 |
| KR2：单图端到端解析耗时 | < 30秒 | 从上传到报告生成全链路计时 |
| KR3：Agent协同任务成功率 | > 95% | Master调度无崩溃、无死锁 |
| KR4：人工复核率 | < 15% | 低置信度标号占比 |
| KR5：用户满意度 (NPS) | > 50 | 工程师月度调研 |

### 2.4 成功指标 KPI

| KPI | 目标 | 说明 |
|-----|------|------|
| ft_strict industrial_score | > 0.55 / > 0.70 | 核心视觉能力指标 |
| ft_full industrial_score | > 0.80 | 含文本提示的基线指标 |
| text_dependency_pct | < 25% | 文本依赖度，越低越好 |
| 标号识别 semantic_f1 | > 0.90 | 语义级别F1 |
| 空间推理 spatial_score | > 0.70 | 空间定位准确性 |
| 推理一致性 consistency | = 1.0 | 文本-图像不矛盾 |
| 知识检索准确率 | > 88% (Top-3) | Knowledge Agent 召回质量 |

### 3.2 用户故事 (User Story)

#### US-001：单图深度解析
> **作为** 机械设计工程师，  
> **我希望** 上传一张专利图纸后，系统自动识别所有标号并输出结构化结果，  
> **以便** 我快速理解图纸结构而无需逐个查阅说明书。

**优先级：** P0  
**关联Agent：** Vision Agent → Report Agent

#### US-002：跨图纸标号一致性校验
> **作为** 知识产权专员，  
> **我希望** 批量导入同一专利的多张附图后，系统自动校验各图的标号连续性与一致性，  
> **以便** 我能快速发现标号错误或遗漏。

**优先级：** P0  
**关联Agent：** Vision Agent → Knowledge Agent → Report Agent

#### US-003：设计合规审查
> **作为** 机械设计工程师，  
> **我希望** 系统基于图纸识别结果自动检查设计是否符合企业标准，  
> **以便** 在正式评审前预先发现合规问题。

**优先级：** P1  
**关联Agent：** Vision Agent → Design Agent → Report Agent

#### US-004：工艺路线自动生成
> **作为** 工艺规划工程师，  
> **我希望** 系统根据图纸中的部件清单和结构关系自动生成工艺路线建议，  
> **以便** 减少重复性的工艺编制工作。

**优先级：** P1  
**关联Agent：** Vision Agent → Process Agent → Report Agent

#### US-005：故障图纸快速定位
> **作为** 设备运维工程师，  
> **我希望** 输入故障代码后，系统自动关联相关图纸并高亮故障部件标号，  
> **以便** 快速定位故障部件进行维修。

**优先级：** P1  
**关联Agent：** Knowledge Agent → Vision Agent → Report Agent

#### US-006：对抗性纠错审核
> **作为** 知识产权专员，  
> **我希望** 系统对比图像识别结果和说明书文字描述，自动发现并标注不一致之处，  
> **以便** 提升专利审核的准确性和效率。

**优先级：** P1  
**关联Agent：** Vision Agent → Knowledge Agent → Master Scheduler（对抗辩论）

#### US-007：低置信度人工复核
> **作为** 任何角色的工程师，  
> **我希望** 系统将置信度低于阈值的标号标红提醒，我能一键确认或修正，  
> **以便** 在AI能力不足时快速介入保障质量。

**优先级：** P0  
**关联Agent：** Vision Agent → 人机协同审核工作台

#### US-008：解析结果推送至PLM
> **作为** 机械设计工程师，  
> **我希望** 审核通过的图纸解析结果自动推送至企业PLM系统，  
> **以便** 解析成果与正式图纸版本绑定，形成可追溯的知识资产。

**优先级：** P2  
**关联系统：** Teamcenter / ENOVIA / 国产PLM

### 4.2 业务流程图

#### 4.2.1 标准解析流程

```
用户上传图纸 (PDF/PNG)
    ↓
Master Scheduler 识别任务类型 = "单图深度解析"
    ↓
[并行调度]
    ├─→ Vision Agent: 输出标号JSON + 空间描述 + 功能推理
    ├─→ Knowledge Agent: 检索相似案例（基于图纸特征向量）
    ↓
[串行依赖]
    ├─→ Design Agent: 基于Vision输出进行设计合规审查
    ├─→ Process Agent: 基于Vision+Design输出进行工艺规划
    ↓
Master Scheduler 聚合结果
    ↓
Report Agent: 生成综合技术报告
    ↓
用户审核 → 确认/修正 → 结论入库
```

#### 4.2.2 对抗性纠错流程（专利审核场景）

```
用户上传专利图纸 + 说明书文字
    ↓
Vision Agent: 基于图像独立识别标号（ft_strict模式）
Knowledge Agent: 提取说明书文字中的标号描述
    ↓
Master Scheduler: 对比 Vision 结果 vs 文字描述
    ↓
若发现不一致（如文字说"2是输出凸缘"，图像判为"主动轴"）
    ↓
触发对抗辩论: Vision Agent 与 Text Agent 各陈述证据
    ↓
人工仲裁: 工程师在UI上查看图像证据与文字证据，一键裁决
    ↓
裁决结果 → 更新企业术语库 + 回流训练数据
```

### 4.3 功能详细说明

#### F-001：智能图纸解析引擎

**功能描述：**
- 支持多格式图纸导入：PDF专利附图、CAD导出图、现场拍照图、扫描图纸
- 自动图纸预处理：去噪、纠偏、分辨率归一化（适配模型 image_max_pixels）
- 多轮对话式识图：用户可追问"标号7和标号8是什么关系？"
- 标号高亮与定位：在原始图上用色块高亮已识别标号，生成可视化 overlay

**Vision Agent 核心能力矩阵：**

| 能力项 | 技术实现 | 评估指标 |
|--------|---------|---------|
| 标号识别 | Qwen2.5-VL-7B 全量微调（蒸馏数据）+ OCR辅助定位 | ft_strict term_f1 > 0.85 |
| 空间定位 | 视觉COT推理（先布局观察 → 再标号定位） | spatial_score > 0.70 |
| 功能推理 | 结合专利背景进行结构-功能关联推理 | functional_score > 0.65 |
| 矛盾检测 | 对抗性纠错模式（文本与图像不一致时以图为准） | consistency = 1.0 |
| 多图关联 | 跨图纸标号一致性校验 | cross_fig_accuracy > 0.90 |

**技术方案（分层策略）：**

| 层级 | 模型 | 用途 | 部署方式 |
|------|------|------|------|
| 教师模型 | GPT-4o / Qwen2.5-VL-72B | 蒸馏数据生成、上限验证 | API调用 |
| 生产模型（推荐） | Qwen2.5-VL-7B 全量微调 | 私有化部署推理 | 5090单卡 vLLM |
| 生产模型（备选） | InternVL2.5-8B 全量微调 | 私有化部署推理 | 5090单卡 vLLM |
| 增强模型（可选） | InternVL2.5-26B | 高精度场景（免微调） | 2×5090 / 1×A100 |
| 辅助组件 | PaddleOCR | 标号文字识别+定位 | CPU |

- 推理加速：vLLM continuous batching，5090 单卡 QPS ~8
- 混合方案：OCR 提取标号文字 → VLM 局部区域理解 → RAG 术语库匹配

**数据策略：**
- 蒸馏数据：由 GPT-4o / Qwen2.5-VL-72B 对 5000-10000 张图纸生成高质量解析结果，经人工审核后作为全量微调训练集
- 数据来源：Google Patents / USPTO / CNIPA 专利附图 + 企业历史图纸（脱敏处理）
- 数据多样性：机械剖面图/爆炸图/装配图/电气图，标号密度3~20+，清晰CAD/扫描件/手机拍照
- 防捷径学习：保留6种模板策略（30%纯视觉 + 10%对抗性 + 15%多轮对话 + 35%标准视觉 + 5%反向 + 5%部分线索）
- 测试监控：每轮评估 ft_full / ft_strict 双轨指标，text_dependency_pct 目标 < 25%

#### F-002：Master Scheduler 主控调度

**功能描述：**
1. **意图识别**：解析用户query，判断需要激活哪些Agent
2. **任务拆解**：将"分析这张图纸"拆解为 [识图→审查→工艺→检索→报告] 的DAG
3. **Agent路由**：根据各Agent负载与专长动态分配任务
4. **冲突仲裁**：当多Agent结论矛盾时，触发对抗辩论机制
5. **人机协同**：当Agent置信度低于阈值时，自动转人工审核

**意图类型映射：**

| 意图类型 | 触发模式 | 必需Agent | 可选Agent |
|---------|---------|----------|----------|
| 单图解析 | "识别图/解析标号/看图" | Vision, Report | Design, Process |
| 批量解析 | "批量图纸/全部解析/专利族" | Vision, Knowledge, Report | — |
| 设计审查 | "审查设计/合规检查/DFM" | Vision, Design, Report | — |
| 工艺规划 | "工艺规划/生成BOM/装配顺序" | Vision, Process, Report | — |
| 知识查询 | "查找类似/历史案例/故障记录" | Vision, Knowledge, Report | — |
| 对抗审核 | "审核附图/纠错标号/不一致" | Vision, Knowledge, Report | — |

#### F-003：Report Agent 报告生成

**功能描述：**
- 多格式输出：Markdown / Word / PDF / 结构化JSON
- 图表自动生成：标号分布图、装配关系图、对比矩阵
- 结论溯源：每个结论标注来源Agent及置信度

#### F-005：人机协同审核工作台

**功能描述：**
- 低置信度标号人工复核：置信度 < 0.8 的标号标红，工程师一键确认或修正
- 多Agent结论对比看板：并列展示 Vision/Design/Process 三Agent的结论，差异项自动高亮
- 一键生成任务单：审核通过的结果可直接推送至PLM/MES生成工艺任务单

#### F-006：Design Agent 设计审查

**功能描述：**
- 标号命名规范性检查（是否符合企业标准术语库）
- 装配关系逻辑校验（如"输入凸缘"是否确实连接"主动轴"）
- 与历史设计图纸比对，识别异常变更

#### F-007：Knowledge Agent (Agentic RAG)

**功能描述：**
- 动态检索：Vision Agent识别出部件后，自动检索企业内所有含该部件的历史图纸
- 闭环反馈：工程师修正结果自动回流为训练数据（需人工审核后入库）
- 多模态知识库：存储图纸图像、标号解析结果、文本描述、三维模型的统一向量表示
- 跨模态检索：以图搜图、以标号搜案例、以结构搜专利
- 知识图谱关联：将标号与企业设备台账、备件库、维修记录关联

### 4.4 交互说明

#### 主要交互流程

1. **上传图纸**：用户通过 Web端 / CAD插件 / 企业微信 / PLM接口 上传图纸
2. **解析进度**：前端通过 SSE 实时展示 COT 推理过程（布局观察→标号识别→功能推理）
3. **结果展示**：在原始图上 overlay 高亮标号，右侧面板展示结构化结果（标号列表、BOM、空间关系）
4. **人工审核**：低置信度标号标红，工程师点击可修改；多Agent差异项自动高亮
5. **报告导出**：一键生成 Word/PDF 技术报告，支持自定义模板
6. **PLM回写**：审核通过后一键推送至PLM系统

### 4.5 边界 & 限制条件

| 条目 | 说明 |
|------|------|
| 图纸格式 | 仅支持 PDF / PNG / JPG / TIFF / DWG（需转换） |
| 图纸分辨率 | image_max_pixels ≤ 262144（约512×512），超过需压缩 |
| 单次批量上限 | 批量解析最大 1000 张/任务 |
| 语言 | 支持中文/英文标号识别，其他语言暂不支持 |
| 模型并发 | 5090 单卡 batch=4，最大并发 8 请求 |

### 4.6 异常处理逻辑

| 异常场景 | 处理策略 |
|---------|---------|
| 图像分辨率过高导致 OOM | 返回错误码 `VISION_OOM`，建议用户压缩至 512×512 以下 |
| Agent 超时（>60s） | 重试 2 次（指数退避），仍失败则标记任务失败并通知用户 |
| 多Agent结论矛盾 | 触发对抗辩论（最多3轮），仍未达成一致则转人工仲裁 |
| Vision Agent 置信度 < 0.8 | 对应标号标红进入人工复核队列 |
| PLM回写失败 | 缓存至Redis，定时重试（最多5次），超时告警 |
| 知识库检索无结果 | 返回"未找到相似案例"，建议用户补充关键词或手动录入 |
| 图纸格式不支持 | 返回错误码 `UNSUPPORTED_FORMAT`，列出支持格式清单 |

## 七、验收标准 【必须】

### 7.1 US-001：单图深度解析

**AC-001-1：标号识别基本功能**
```
Given 用户上传一张有效的专利图纸（PDF/PNG格式）
When  Vision Agent 完成解析
Then  返回结构化 JSON，包含所有可见标号的名称、位置、置信度
And   ft_strict industrial_score > 0.55
And   响应时间 < 30秒
```

**AC-001-2：标号高亮可视化**
```
Given Vision Agent 已完成标号识别
When  前端渲染结果
Then  原始图纸上以色块 overlay 高亮每个已识别标号
And   置信度 > 0.8 的标号显示绿色，< 0.8 的显示橙色/红色
```

**AC-001-3：COT推理过程流式展示**
```
Given 用户开启 SSE 流式解析
When  Vision Agent 执行推理
Then  前端实时展示推理步骤（布局观察→标号定位→功能推理）
And   首字节延迟 < 500ms
```

### 7.2 US-002：跨图纸标号一致性校验

**AC-002-1：批量导入**
```
Given 用户批量上传同一专利的 N 张附图（N ≤ 1000）
When  系统完成批量解析
Then  返回每张图的标号识别结果
And   自动生成跨图标号一致性报告
And   cross_fig_accuracy > 0.90
```

**AC-002-2：标号连续性检测**
```
Given 多张图已完成解析
When  Knowledge Agent 执行跨图关联
Then  标出各图中相同标号的名称是否一致
And   标出跨图缺失或新增的标号
```

### 7.3 US-003：设计合规审查

**AC-003-1：规范性检查**
```
Given Vision Agent 已输出标号结果
When  Design Agent 执行设计审查
Then  输出合规性报告，列出每个标号是否符合企业标准术语库
And   识别装配关系逻辑错误
And   标注与历史设计图的异常变更
```

### 7.4 US-006：对抗性纠错审核

**AC-006-1：矛盾检测**
```
Given 用户上传专利图纸 + 说明书文字
When  Vision Agent（图像独立识别）与 Knowledge Agent（文字提取）结果不一致
Then  Master Scheduler 自动检测到冲突
And   标注冲突标号及双方判断结果
```

**AC-006-2：对抗辩论**
```
Given 检测到命名冲突（置信度差距 > 15%）
When  触发对抗辩论机制
Then  Vision Agent 和 Knowledge Agent 各出具证据陈述（最多3轮）
And   若置信度 > 85% 自动裁决，否则转人工仲裁
```

### 7.5 US-007：低置信度人工复核

**AC-007-1：复核流程**
```
Given Vision Agent 识别结果中有标号置信度 < 0.8
When  结果展示在审核工作台
Then  低置信度标号以红色标记
And   工程师可一键确认或修改名称
And   修正结果自动回流为训练数据候选（需二次审核）
```

### 7.6 US-008：PLM回写

**AC-008-1：Teamcenter回写**
```
Given 工程师在审核工作台确认解析结果
When  点击"推送至PLM"
Then  解析结果写入 Teamcenter ItemRevision 的自定义属性
And   若图纸处于审核中，自动推送AI识图结论至审核人 Workflow
And   回写失败时自动重试（最多5次），超时告警
```

## 九、范围外 (Out of Scope) 【防坑】

### 9.1 明确不做的功能

| 功能 | 原因 |
|------|------|
| 三维CAD模型实时解析 | 当前模型仅支持2D图纸图像，三维模型解析技术不成熟 |
| 自动生成CAD图纸 | 系统定位为"识图"而非"画图"，生成图纸超出Vision Agent能力范围 |
| 替代工程师决策 | 系统定位为"硅基实习生"，最终决策权始终保留给工程师 |
| 公有云部署 | 工业图纸涉密，仅支持私有化部署 |
| 非工业图纸场景 | 不支持建筑/电路板PCB/地图等非机械工业图纸类型（MVP阶段） |
| 实时视频流识别 | 不支持摄像头实时视频流的图纸识别，仅支持静态图像 |

### 9.2 后续版本规划

| 版本 | 预计时间 | 规划功能 |
|------|---------|---------|

### 9.3 已知遗留问题

| 问题 | 影响 | 临时方案 |
|------|------|---------|
| ft_strict 当前仍偏低 | 纯视觉模式下识别准确率不足 | 持续投入防捷径学习数据建设，ft_strict < 0.4 时冻结功能升级 |
| text_dependency 偏高 | 模型过度依赖文本提示 | 增加纯视觉训练数据比例（当前30%→目标50%） |
| 多Agent协调开销 | 调用链长可能导致延迟 | 层级式调度为主，仅冲突时触发平等协商；设置Agent调用预算上限 |
| PLM适配器开发量大 | 各厂商API差异显著 | 优先REST API通用接口 + 标准化中间件适配器 |

### 10.3 竞品分析

|-----------|------|------|------------|
| 传统OCR+规则引擎 | 成熟稳定、成本低 | 无法理解图纸语义，仅能识别文字 | 基于VLM的结构推理+功能理解 |
| 通用大模型（GPT-4o等） | 通用能力强 | 成本高、无法私有部署、延迟大 | 大模型做蒸馏教师，小模型私有部署推理 |
| 传统PLM内置识别 | 与系统紧耦合 | 功能单一、不支持跨系统协同 | 多Agent协同+跨系统适配器 |
| 人工+Excel | 灵活 | 效率低、不可扩展、知识沉淀差 | 自动化解析+知识库沉淀 |

### 10.4 风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| Vision Agent 视觉能力不足 | 系统根基动摇 | 持续投入防捷径学习数据建设，ft_strict < 0.4 则冻结功能升级 |
| 多Agent协调开销过高 | 响应延迟、Token成本飙升 | 层级式调度为主，冲突时才触发平等协商；设置调用预算上限 |
| 企业数据安全顾虑 | 图纸涉密无法上云 | 全私有化部署，5090本地服务器，数据不出厂区 |
| PLM系统对接复杂 | 集成周期长 | 优先REST API通用接口，提供标准化中间件适配器 |
| 工程师抵触AI替代 | 采纳率低 | 定位为"硅基实习生"而非替代者，强调人机协同与决策权保留 |
